# Project Guidance for Claude Code

This file holds always-on rules so you don't have to repeat them every prompt.
Read it at the start of every session and follow it for all work in this repo.

## Stack

- **Backend:** Java 17 (Gradle toolchain, see `build.gradle`), Spring Boot 3.5.x
- **Build tool:** **Gradle** with the Gradle Wrapper — use `./gradlew` (Linux/macOS)
  or `gradlew.bat` (Windows PowerShell: `.\gradlew.bat`). Never a globally
  installed `gradle`. There is **no Maven** in this project.
- **Database:** MySQL 8.x — this is important. Do NOT default to PostgreSQL or H2
  for application config. H2 is acceptable ONLY in tests (or use Testcontainers
  with MySQL for integration tests).
- **Lombok** is used (`@Getter`/`@Setter`/etc.); the annotation processor is
  configured in `build.gradle`.
- **Mail:** Spring Boot Mail starter (overdue-borrow reminder emails).
- Config lives in `src/main/resources/application.properties` (properties, not YAML).

### Common commands

```bash
./gradlew build                                   # compile + test
./gradlew test                                    # run all tests
./gradlew test --tests "*TimeStampCodeGeneratorTest"   # run one test class
./gradlew bootRun                                 # run the app
```

On Windows set `JAVA_HOME` to a JDK 17 before running, e.g.
`C:\Users\<you>\.jdks\ms-17.0.19`.

## How we work

1. **Plan before building.** For any non-trivial task (3+ steps or an architectural
   decision), enter plan mode and show the plan before writing code. Wait for
   confirmation on anything that creates new modules, changes the schema, or
   touches security.
2. **Verify by testing.** After implementing, write tests and run `./gradlew test`.
   Confirm they pass before declaring the task done. If tests fail, read the stack
   trace, fix the cause, and re-run the full suite — don't paper over failures.
3. **Keep it simple.** Prefer the simplest design that works. After writing code,
   review it for overengineering and remove anything speculative.
4. **Use the skills in `.claude/skills/`.** When a task matches a skill's
   description (JPA, Spring config, testing, etc.), consult that skill.

## Backend conventions

- **Never expose JPA entities directly in API responses or accept them as request
  bodies.** Always map to/from DTOs (request DTO in, response DTO out).
- Use **constructor injection** (no field injection / no `@Autowired` on fields).
- Layered structure (actual package layout under
  `com.example.m6_thermal_power_plant_api`):
  ```
  controller/   REST endpoints — thin, no business logic (grouped by domain:
                auth, equipment, maintenance, consumable, spare_part, tool)
  service/      business logic, transactions (@Transactional here, not on controllers;
                impl/, soft_delete/, and per-domain subpackages)
  repository/   Spring Data JPA interfaces
  entity/       @Entity classes (never returned from controllers; base/, enums/, tool/)
  dto/          request/response DTOs (per-domain, often request/ + response/ subfolders)
  config/       configuration classes
  security/     security-related classes
  exception/    custom exceptions + a @RestControllerAdvice global handler
  util/         helpers (e.g. TimeStampCodeGenerator, UniqueCodeRetryExecutor)
  ```
- DTOs in this project are plain classes (with Lombok), not records — follow the
  existing style when adding new ones.
- Validate request DTOs with Jakarta Bean Validation (`@Valid`, `@NotNull`, etc.).
- Return proper HTTP status codes; handle errors in one `@RestControllerAdvice`,
  not scattered try/catch.

## Business code generation

- Business codes (department_code, request_code, order_code, consumable_code, …)
  are generated by `util/TimeStampCodeGenerator`: format `PREFIX-yyMMddHHmmss-SEQ`,
  with a per-prefix in-memory counter for the 3-digit SEQ.
- The in-memory counter is a best-effort de-dup, not a guarantee. Treat the DB
  unique constraint (see `db/upgrade-soft-delete-unique-constraints.sql`) as the
  final guard, and wrap saves with `util/UniqueCodeRetryExecutor` to regenerate +
  retry on `DataIntegrityViolationException`.

## Database conventions (MySQL)

- Schema is currently managed by Hibernate via
  `spring.jpa.hibernate.ddl-auto=update`. Supplementary SQL scripts live in
  `src/main/resources/db/` (sample data, soft-delete unique constraints). **There
  is no Flyway** in this project yet — if you introduce migrations, add the Flyway
  dependency and switch `ddl-auto` to `validate`/`none` first.
- Use `utf8mb4` charset and a `utf8mb4_unicode_ci` collation.
- Table and column names in `snake_case`; map with explicit `@Column` /
  `@Table` names so Java camelCase and SQL snake_case stay decoupled.
- Prefer `BIGINT` auto-increment primary keys (`@GeneratedValue(strategy = IDENTITY)`)
  unless there's a reason to use UUIDs.
- Be mindful of the N+1 query problem — see the persistence-jpa skill.

## Security

- Don't commit secrets. Use environment variables or an untracked
  `application-local.properties`. The committed `application.properties` should
  reference `${ENV_VAR}` placeholders (e.g. `${MAIL_USERNAME:}`); do not commit the
  real DB password or mail password.
- Validate and sanitize all external input. Rely on parameter binding / JPA, never
  string-concatenated SQL.
